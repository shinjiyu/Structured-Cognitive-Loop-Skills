# 设计背景与动机

## 从一次性触发到连续运行

传统 AI Agent 的运行形态是**被动响应**：用户发送指令，Agent 回复，会话结束。这种形态有一个隐含假设——**每次交互都是独立的**，Agent 不需要在会话之间保持目标、追踪进度或自我调整。

当我们把 Agent 升级为**定时/连续运行**时（如 OpenClaw 的 Heartbeat 机制），这个假设彻底失效：

- Agent 需要在无人干预的情况下，持续多天、多周执行一个长期目标
- 每次心跳都是一个独立的 Agent Loop，但它们共同服务于同一个持久目标
- 错误会跨轮次积累，良好的行为也需要跨轮次强化

这种运行形态带来了两类新问题：

---

## 问题一：错误积累

**症状**：小的认知偏差或执行错误在每一轮中被当作正常输入，到下一轮时进一步放大，形成负向正反馈，最终导致系统行为完全偏离预期。

**已有应对**：  
- 结构化调试协议（有限次数后强制回退）  
- 范围守卫（每步保持可运行状态）  
- 交叉进化框架（归纳 Agent 发现缺陷，进化 Agent 修复）  

错误积累问题通过**自修复机制**可以得到较好的控制。

---

## 问题二：输入枯竭与空转

**症状**：系统只在既有的技能、概念和流程内循环，没有新领域、新概念、新任务类型注入，产出价值递减，最终进入空转——每轮心跳都在重复同一套动作，没有实质进展。

**这是更难解决的问题**，因为它不是「做错了」，而是「做的是正确的事，但世界已经不需要这些了」或「Agent 的能力边界限制了它能做的事」。

**应对方向**：
1. **主动学习**：主动从外部（网页、文档、用户对话）发现新概念，扩展认知边界
2. **能力自举**：当发现能力缺口时，自动派生新技能，而不是放弃或请求用户干预

---

## 问题三：抽象目标的执行

当用户给出的是高级抽象目标（如「帮我运营小红书」「让我的产品获得更多用户」），而**不再提供任何后续指令**时，Agent 面临的挑战是：

**搜索空间爆炸**：「运营小红书」的第一步应该是什么？内容策略？竞品分析？账号设置？数据分析？每个选择都引出无数子选择，Agent 在第一步就可能陷入决策瘫痪。

**传统「先规划再执行」的失效**：
| 假设 | 在多轮场景下的现实 |
|------|-------------------|
| 有人审阅计划 | 无人在旁，必须自主决策 |
| 一次执行完毕 | 需要数周、数月持续执行 |
| 环境保持不变 | 每轮之间环境已发生变化 |
| 计划不会过期 | 第 10 轮执行第 1 轮的计划，往往已经过时 |

**正确的范式**：不是「制定计划再执行」，而是「每轮感知当前状态，决策当前最优单步行动」。这是 OODA Loop、ReAct 等框架的核心思想，也是本系统的出发点。

---

## 解决方案的组合

本系统选择了三个互补框架的组合：

### SCL（Structured Cognitive Loop）

来源：Myung Ho Kim, 2025 ([arxiv:2510.15952](https://arxiv.org/abs/2510.15952))

SCL 将 Agent 的认知过程显式分离为五个模块（R-CCAM），解决「每步如何可靠执行」的问题：
- **Retrieval**：从记忆中准确获取当前所需上下文
- **Cognition**：LLM 推理，在有限搜索空间内做决策
- **Control**：执行前验证，防止越界行为，实现零策略违规
- **Action**：工具执行
- **Memory**：结构化写回，确保状态跨轮次持久

SCL 不解决「该做什么」，它解决「做的过程中如何保持纪律」。

### ReCAP（Recursive Context-Aware Reasoning and Planning）

来源：Stanford + MIT, NeurIPS 2025 ([arxiv:2510.23822](https://arxiv.org/abs/2510.23822))

ReCAP 解决「搜索空间爆炸」问题：
- 将抽象目标分解为结构化子任务树（折叠搜索空间）
- 执行第一个子任务后，动态修正剩余任务树（适应变化）
- 通过「父计划再注入」确保执行细节时不丢失顶层目标

ReCAP 负责「该做什么」，SCL 负责「怎么做好」。两者协作：

```
抽象目标
  ↓ ReCAP：折叠为任务树，决定「当前子任务」
当前子任务
  ↓ SCL：可靠执行当前子任务
执行结果
  ↓ 反馈给 ReCAP：树如何修正？
```

### 能力缺口自举（Capability Gap Bootstrapping）

本系统提出的原创设计，解决「Agent 在执行中发现自己缺少某类工具」时的自动扩展问题：

**核心原理**：Agent 天然具备一组「原子能力」（web_search + write_file + shell_exec + LLM）。当发现能力缺口时，不请求用户干预，而是用原子能力派生新 Skill。

**自举路径**：
```
缺少画图能力
  → 用 web_search 搜索画图 API / Skill
  → 用 write_file 写入新 Skill 文件
  → 下一次心跳，新 Skill 自动生效
  → 继续原任务
```

多次 ad-hoc 派生后，active-learning 会将重复的派生流程固化为正式 Skill，形成良性进化循环。

## 三者的分工

```
OpenClaw Heartbeat（无限循环引擎）
  │
  ├── ReCAP（宏观）：抽象目标 → 当前应执行的子任务
  │     「搜索空间折叠，决定走哪条路」
  │
  ├── SCL R-CCAM（微观）：子任务的可靠执行
  │     「记忆可靠，行动有纪律，状态持久」
  │
  └── 能力缺口自举（元层）：发现边界时扩展边界
        「用原子能力派生新能力，无需人工干预」
```

这三者叠加，使 Agent 能够：
1. 接受一句抽象目标，自主分解执行
2. 在长期运行中保持行为纪律，不积累错误
3. 在遇到能力边界时自动扩展，而不是停下来等待
